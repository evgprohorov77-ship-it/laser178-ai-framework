# RFC Process

## Версия

1.0.0

## Назначение

Любое архитектурное изменение, новая возможность или новый Engine сначала оформляются как RFC (Request for Comments). Только после одобрения RFC разрешается реализация.

## Когда нужен RFC

- Новый Engine.
- Новый адаптер (WordPress, GitHub, Cloudflare, SMTP и т.д.).
- Изменение Decision Engine.
- Изменение Policy.
- Изменение структуры репозитория.
- Изменение Capability.
- Изменение Production Policy.
- Любое изменение, которое влияет на безопасность.

## Не требует RFC

- Исправление опечатки в документации.
- Обновление Knowledge.
- Новый unit-test.
- Локальный эксперимент, не влияющий на архитектуру.

## Процесс

```
Draft → Review → Approved → Implemented → Merged
```

1. Автор создаёт RFC в `RFC/NNNN-title.md`.
2. RFC получает статус `Draft`.
3. Проводится review (владелец + архитектор).
4. RFC переходит в `Approved` или `Rejected`.
5. После `Approved` начинается реализация.
6. После реализации RFC переходит в `Implemented`.

## Статусы

- `Draft` — черновик, обсуждается.
- `Proposed` — готов к review.
- `Approved` — одобрен, можно реализовывать.
- `Rejected` — отклонён.
- `Implemented` — реализован и влит в main.
- `Superseded` — заменён новым RFC.

## Шаблон

См. `RFC/template.md`.

## Нумерация

RFC нумеруются четырёхзначными числами: `0001`, `0002` и т.д.
