# Authorization Policy

## Версия

1.0.0

## Цель

Определить, кто и при каких условиях может выполнять действия в LAOS.

## Принцип

Authorization Engine отвечает только на один вопрос: **«Можно ли выполнить действие?»**  
Как выполнить — это задача Action Engine.

## Уровни авторизации

| Уровень | Кто | Что может |
|---------|-----|-----------|
| owner | Владелец сайта | Подтверждать P0/P1, изменять политики, включать/отключать capability. |
| admin | Hermes + администратор | Читать, предлагать, выполнять P2/P3 с autofix. |
| agent | Hermes | Только то, что разрешено Capability System. |
| readonly | Гость/аудитор | Только аудит, без изменений. |

## Правила по умолчанию

1. В production автоматические изменения запрещены, если не включён explicit override.
2. P0 и P1 требуют подтверждения владельца в любой среде.
3. P2/P3 разрешены с autofix только в development/staging и при confidence ≥ 0.70.
4. Отсутствие capability = отказ.
5. Отсутствие environment = отказ.
6. Отсутствие successful backup = отказ.

## Capability-based authorization

Каждый запрос на действие проверяется по списку:

```
- Есть ли capability для этого действия?
- Разрешено ли действие в текущем environment?
- Требуется ли approval?
- Есть ли одобренный approval?
- Прошёл ли Dry Run?
- Есть ли успешный backup?
- Не нарушает ли действие production lock?
```

## Decision

Authorization Engine возвращает:

- `allowed: true` — действие разрешено.
- `allowed: false` — действие запрещено с обязательным `reason`.

## Аудит

Каждый запрос авторизации логируется в `Logs/authorization/`.
