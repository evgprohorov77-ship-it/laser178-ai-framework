# Rollback Policy

## Цель

Определить, как откатывать автоматические изменения, если Verification Engine не прошёл.

## Принцип

Любое автоматическое исправление должно быть обратимо. Если откат невозможен — autofix запрещён.

## Классификация изменений

| Тип изменения | Rollback возможен | Способ |
|---------------|-------------------|--------|
| Title | Да | Вернуть старый title из backup или CMS |
| Description | Да | Вернуть старый description из backup или CMS |
| H1 | Да | Изменить шаблон или CSS |
| ALT | Да | Вернуть старый alt или оставить пустым |
| CSS/JS | Да | Восстановить файл из backup |
| Canonical | Да | Удалить добавленный тег |
| Удаление плагина | Да (сложно) | Требуется backup БД + файлов |
| Смена темы | Нет | **Запрещено автоматически** |
| Изменение URL | Нет | **Запрещено автоматически** |
| Изменение DNS | Нет | **Запрещено автоматически** |

## Успешный откат

Откат считается успешным, если:

1. Страница открывается (HTTP 200).
2. Нет новых ошибок.
3. Проблема, вызванная автоматическим изменением, устранена.
4. Состояние сайта не хуже, чем до изменения.

## Процедура отката

1. **Before**: Action Engine делает snapshot затронутых файлов/записей.
2. **Apply**: Action Engine применяет изменение.
3. **Verify**: Verification Engine проверяет результат.
4. **On fail**: Action Engine восстанавливает snapshot.
5. **Re-verify**: Verification Engine проверяет, что откат успешен.
6. **Log**: Logger записывает все шаги.
7. **Issue**: Если откат не удался — создаётся P0 Issue.

## Резервное копирование

- CSS/JS: backup-плагин `l178_backup_plugins.php`.
- WordPress записи: экспорт через WP-CLI или REST API.
- База данных: инструменты хостинга (Beget).
