# Backup Policy

## Версия

1.0.0

## Цель

Перед любым изменением убедиться, что его можно откатить.

## Типы backup

| Тип | Что копируется | Когда |
|-----|----------------|-------|
| full | База данных + файлы сайта | Раз в неделю или перед массовыми изменениями. |
| incremental | Изменённые файлы | Перед каждым autofix. |
| plugin | `/wp-content/plugins/` | Перед изменениями плагинов. |
| content | Конкретная страница/пост | Перед edit_title, edit_description и т.д. |
| config | `wp-config.php`, `.htaccess` | Перед изменениями конфигурации. |

## Требования к backup Engine

1. Backup не использует PHP, WordPress, FTP.
2. Backup — это алгоритм с явными шагами.
3. После создания backup проверяется его целостность.
4. Только после успешной проверки разрешается изменение.
5. Каждый backup получает UUID и связан с approval UUID.

## Алгоритм Backup Engine

```
1. Получить target (файл, страница, БД).
2. Сгенерировать backup UUID.
3. Создать копию через SFTP/SSH/резервную копию хостинга.
4. Проверить хеш-суммы исходного и копии.
5. Записать метаданные: UUID, timestamp, target, hash, approval UUID.
6. Вернуть backup ID в Authorization Engine.
7. Только после этого Action Engine может работать.
```

## Срок хранения

- Инкрементальные backup — 30 дней.
- Full backup — 90 дней.
- После успешного verification backup перемещается в архив.

## Проверка backup

Backup считается успешным, если:

- файл/данные существуют;
- хеш совпадает;
- размер больше 0;
- копию можно прочитать.

## Без backup — нет действия

Authorization Engine автоматически отказывает в любом действии без успешного backup.
