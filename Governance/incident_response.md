# Governance: Incident Response

## Статус

- **Действует с:** LAOS-005
- **Автор:** Owner
- **Цель:** заранее определить действия при инцидентах.

## Уровни инцидентов

| Уровень | Описание | Пример | Реакция |
|---------|----------|--------|---------|
| P0 — Critical | Сайт недоступен, данные повреждены, утечка | Сайт упал, база повреждена | Немедленно, owner notification в течение 15 минут |
| P1 — High | Важная функция сломана, нет доступа | REST API не отвечает, проверка не прошла | В течение 1 часа |
| P2 — Medium | Некритичная ошибка, обходной путь есть | Backup повреждён, но есть другая копия | В течение 4 часов |
| P3 — Low | Косметическая проблема | Опечатка, неработающая ссылка | В рабочем порядке |

## Сценарии и действия

### 1. Сайт упал

1. Operator или Hermes проверяет доступность (curl, браузер, логи хостинга).
2. Если проблема на хостинге — связываемся с поддержкой Beget.
3. Если проблема после наших изменений — немедленный откат последнего изменения.
4. Восстановление из последнего рабочего backup.
5. Owner уведомляется в течение 15 минут.
6. Запись в `Audit/incidents/` и `Changelog/`.
7. Post-mortem в течение 24 часов.

### 2. Пропали статьи

1. Проверить, не отключён ли плагин, не сброшены ли настройки.
2. Проверить последние backup базы данных и файлов.
3. Если данные есть в backup — восстановить.
4. Если нет backup — проверить локальные черновики в `Content/`.
5. Не пытаться восстановить автоматически без owner approval, если причина неизвестна.
6. Owner уведомляется немедленно.

### 3. База данных повреждена

1. Остановить любые записи в базу.
2. Проверить целостность backup:
   - `Backup/latest/`;
   - хостинговые backup (Beget);
   - локальные копии.
3. Если backup валиден — восстановить.
4. Если backup повреждён — попробовать предыдущие версии.
5. Если нет валидного backup — сообщить Owner и разработать план восстановления вручную.
6. Никакие production-изменения до полного восстановления.

### 4. REST API не отвечает

1. Проверить статус API (curl, status page).
2. Проверить сетевые настройки, токены, rate limits.
3. Если это WordPress REST API — LAOS не использует его в production в v1.x. См. `RFC/0002-wordpress-adapter.md`.
4. Если проблема критична — отключить интеграцию и перейти в ручной режим.
5. Owner уведомляется.

### 5. Проверка не прошла (validate_knowledge.py, check_content.py)

1. Не применять изменения.
2. Записать ошибки в `Audit/validation_failures/`.
3. Hermes анализирует ошибки и предлагает исправления.
4. После исправления повторно запустить проверки.
5. Только после успешной проверки — owner approval и release.

### 6. Backup повреждён

1. Проверить предыдущие версии backup (rotation).
2. Проверить целостность файлов (checksum, хеши).
3. Попробовать восстановить из другого источника (Beget, Git, локальные копии).
4. Если все backup повреждены — немедленно сообщить Owner.
5. Создать новый backup вручную после восстановления.

## Общие правила инцидентов

- **Не паниковать.** Действовать по процедуре.
- **Не делать хуже.** Любое действие в production при инциденте требует owner approval, кроме явно разрешённых в `Governance/incident_response.md`.
- **Фиксировать всё.** Логи, скриншоты, временные метки, действия.
- **Owner в курсе.** Owner уведомляется обо всех P0/P1 и любых инцидентах с данными.
- **Post-mortem.** После P0/P1 — обязательный разбор в течение 24 часов.

## Контакты при инциденте

- Owner: Telegram DM с Евгением Прохоровым
- Хостинг: Beget (support через личный кабинет)
- Operator: назначается Owner

## Регулярные проверки

- Раз в неделю: проверка backup.
- Раз в месяц: тестовое восстановление из backup.
- Раз в квартал: review `Governance/incident_response.md`.
